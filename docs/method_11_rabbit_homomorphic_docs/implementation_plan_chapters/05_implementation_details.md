## 5. 実装計画と管理 📋

### ⚠️ 実装における絶対禁止事項

以下の実装パターンは、たとえ短期的に機能しても、システムの核心的セキュリティを損なうため**絶対に禁止**します：

1. `decrypt(encrypted_data, key, is_true_file=True)` のような、鍵以外のパラメータによる復号経路の決定
2. 暗号ファイル内への `"true_section"` や `"false_section"` などの識別子の埋め込み
3. 鍵データ内への経路情報の直接埋め込み（例: `{"key": "...", "type": "true"}` など）
4. 一方の鍵から他方の鍵を導出または推測可能な実装（例: `false_key = true_key[::-1]` など）
5. ソースコード内の明示的な分岐による復号経路の決定
6. 共通鍵導出元（シード値）の使用による関連性のある鍵生成
7. 暗号化時・復号時に経路識別子をパラメータとして受け渡す API 設計
8. ソースコード難読化による安全性の確保（解析耐性は難読化ではなく数学的に保証すること）

上記の禁止事項に違反する実装は、テストに合格しても**即時拒否**され、実装のやり直しが必要となります。真に安全な実装は、鍵の数学的性質のみによって経路を決定し、ソースコードを完全公開しても安全である必要があります。

### 暗号システムの絶対要件

本システムの実装において、以下の要件は絶対に譲れない核心的セキュリティ原則です：

1. **鍵のみによる文書区別の原則**：

   - 暗号文書の区別は鍵の違いのみによって行われなければならない
   - ファイル内のフラグや識別子による区別は絶対に禁止
   - メタデータやヘッダー情報に経路情報を含めてはならない
   - **重要**: 両方の鍵は実装上完全に等価で、システム内では「正規/非正規」の区別が一切存在しない

2. **鍵の交差推測不可能性**：

   - 鍵 A から鍵 B を数学的に導出・推測することが不可能であること
   - 一方の鍵の漏洩が他方の鍵のセキュリティに影響しない設計

3. **ソースコード解析耐性**：

   - ソースコードの完全な解析によっても、鍵なしでの復号が不可能であること
   - ソースコードの難読化に依存しない数学的安全性の保証

4. **ファイル区画分離回避**：
   - 真偽情報を別々のファイル区画に格納する設計の回避
   - 単一の暗号化ストリームに両方の情報を融合して保存する設計

これらの要件は、タスク計画や進捗状況に関わらず常に最優先されるべき原則であり、どのような設計変更や最適化を行う場合でも必ず遵守しなければなりません。

### 最適化された実装サイクル構成

スケルトンファースト戦略と継続的結合テストに基づき、以下のように実装サイクルを再構成しました。各サイクルは明確な成果物と検証ポイントを持ち、サイクルごとにメインストリームへの統合を行います。

#### スケルトンファースト実装戦略

この戦略に基づき、開発初期からエンドツーエンドの基本フローを確立します：

1. サイクル 1 で基本的な暗号化・復号の骨格を実装
2. 内部機能はプレースホルダーまたは単純な実装（XOR 暗号など）で代替
3. 将来実装される全機能のインターフェースを事前に定義
4. 各サイクルで機能を段階的に実装し、プレースホルダーを実際の実装に置き換え

これにより、「常に動く」原則を維持し、早期から結合テストが可能になります。

#### メインフレーム統合モデル

各サイクルの成果物は「メインフレームに統合可能な状態」をゴールとし、継続的な統合とテストを実現します：

1. 各サイクル終了時に `rabbit_homomorphic.py`（メインフレーム）への統合テストを実施
2. 統合できない実装は「未完了」と見なし、次のサイクルに進まない
3. 統合テストの自動化スクリプトを早期に実装
4. サイクル間の依存関係を明確に管理

### 継続的な結合テスト戦略

各サイクルの実装が完了したら、以下の手順で継続的な結合テストを実施します：

1. **単体テスト**: 各コンポーネントの単体テストを実行
2. **サイクル内結合テスト**: サイクル内のコンポーネント間の結合テストを実行
3. **メインフレーム統合テスト**: 実装したコンポーネントをメインフレーム(`rabbit_homomorphic.py`)に統合し、既存機能との互換性を検証
4. **回帰テスト**: 既存の機能が引き続き正常に動作することを確認
5. **エンドツーエンドテスト**: 暗号化 → 復号の完全なフローテストを実行

これにより、「単体は動くが全体が動かない」というリスクを最小化し、実装の早い段階から問題を発見・修正することが可能になります。
